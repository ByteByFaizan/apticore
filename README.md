## About

**AptiCore** is an AI-powered recruitment platform that eliminates unconscious bias from the hiring process. It anonymizes candidate data, evaluates skills using Google's Gemini AI, and provides transparent, explainable rankings — ensuring every candidate is judged on **merit alone**.

Built for the **Google Solution Challenge 2026**, AptiCore demonstrates how AI can be used responsibly to promote fairness and equity in employment.

---

## The Problem

Traditional hiring processes are rife with unconscious bias:

- **Name bias** — Candidates with "foreign-sounding" names receive 30% fewer callbacks
- **Gender bias** — Identical resumes with male names are rated higher than those with female names
- **Institution bias** — Ivy League graduates are preferred regardless of actual skill fit
- **Ethnicity bias** — Racial minorities face systematic disadvantages in resume screening

AptiCore solves this by **removing all personally identifiable information (PII) before evaluation** and using AI to assess candidates purely on their skills, experience, and job-requirement alignment.

---

## Key Features

### Blind Evaluation
Strips names, emails, phone numbers, photos, addresses, and institution names before any AI analysis begins. Candidates become `Candidate Alpha-1`, `Beta-3`, etc.

### AI-Powered Skill Matching
Uses a **hybrid matching engine** combining keyword extraction with semantic similarity (via Gemini embeddings) to evaluate how well each candidate's skills align with the job description.

### Bias Detection & Reporting
Runs a **dual-pipeline comparison** — simulating how a biased traditional system *would have* ranked candidates vs. how AptiCore's fair system ranks them — producing quantified fairness scores with before/after deltas.

### Explainable AI
Every ranking comes with a plain-English explanation generated by Gemini, detailing *why* a candidate scored the way they did. No black boxes.

### Multi-Format Resume Parsing
Accepts **PDF**, **DOCX**, and **TXT** resumes. PDFs are parsed using **Mozilla PDF.js** (`pdfjs-dist`) with position-aware text extraction, multi-column layout detection, and a full text cleaning pipeline. Structured data (skills, experience, education) is then extracted using Gemini AI with **Zod schema validation**.

### Enterprise-Grade Security
Multi-layer defense: Edge CORS, CSRF protection, token-bucket rate limiting, Zod input validation, and strict Firestore/Storage security rules.

## Getting Started

### Prerequisites

- **Node.js** >= 18.17
- **npm** >= 9.0
- A [Firebase project](https://console.firebase.google.com/) with Auth, Firestore, and Storage enabled
- A [Google AI Studio](https://aistudio.google.com/) API key for Gemini

### Installation

```bash
# 1. Clone the repository
git clone https://github.com/ByteByFaizan/apticore.git
cd apticore

# 2. Install dependencies
npm install

# 3. Set up environment variables
cp .env.example .env.local
# Edit .env.local with your credentials (see below)

# 4. Start the development server
npm run dev
```

The app will be available at `http://localhost:3000`.

### Available Scripts

| Command | Description |
|---------|-------------|
| `npm run dev` | Start development server with hot reload |
| `npm run build` | Create optimized production build |
| `npm run start` | Start production server |
| `npm run lint` | Run ESLint checks |

## AI Pipeline

When a batch is processed, it flows through a **9-step automated pipeline** using Next.js `after()` for background execution:

```
Step 1   PARSING              → Extract JD requirements (Gemini 2.5 Pro, temp=0)
 Step 2   PARSING (resumes)    → PDF.js / JSZip / UTF-8 text extraction + cleaning
                                 → Gemini 2.5 Pro structured resume parse (Zod validated)
 Step 3   ANALYZING_BIAS_BEFORE → Simulated traditional (biased) hiring pipeline
 Step 4   ANONYMIZING          → PII stripped, deterministic seeded shuffle (DJB2 hash)
 Step 5   MATCHING             → Hybrid keyword (70%) + semantic (30%) scoring
 Step 6   RANKING              → Candidates ranked by composite score
 Step 7   EXPLAINING           → Gemini 2.5 Flash generates human-readable explanations
 Step 8   ANALYZING_BIAS_AFTER → Fair pipeline comparison & fairness scoring
 Step 9   COMPLETE             → Results saved with rankings, explanations & bias report
```

### AI Models Used

| Model | Purpose | Configuration |
|-------|---------|---------------|
| **Gemini 2.5 Pro** | Resume parsing, JD extraction | `temperature: 0`, `topK: 1`, `topP: 1`, JSON mode, Zod validated |
| **Gemini 2.5 Flash** | Candidate explanations | `temperature: 0`, `topK: 1`, `topP: 1`, `maxTokens: 800` |
| **gemini-embedding-001** | Semantic skill similarity | 768-dimensional vectors |

### Matching Strategy

The hybrid matcher combines two approaches:

1. **Keyword Matching (70%)** — O(1) skill overlap via normalized keyword sets
2. **Semantic Similarity (30%)** — Cosine similarity between Gemini embeddings of candidate skills and JD requirements

Final score = `(keywordScore × 0.7) + (semanticBoost × 0.3)` — keyword precision dominant with semantic understanding as a boost.

### Determinism Guarantees

The pipeline is **fully deterministic** — same input always produces the same output:

| Source | Fix |
|--------|-----|
| AI randomness | `temperature: 0`, `topK: 1`, `topP: 1` (greedy decoding) |
| Candidate shuffle | Seeded PRNG (Mulberry32) with DJB2 hash of `batchId` |
| Retry timing | Fixed 250ms jitter (no `Math.random()`) |
| AI output format | Zod schema validation with retry-on-invalid-JSON |
| Resume parsing | Position-aware PDF.js extraction, deterministic text cleaning |

## Security

AptiCore implements defense-in-depth with multiple security layers:

| Layer | Mechanism | Details |
|-------|-----------|---------|
| **Edge** | CORS Middleware | Origin allowlist enforced at edge; blocks unauthorized origins |
| **Transport** | CSRF Protection | `X-Requested-With` header required on all mutations |
| **Rate Limiting** | Token Bucket | Per-IP/user limits with RFC-compliant `RateLimit-*` headers |
| **Validation** | Zod Schemas | All request bodies validated against strict schemas |
| **Auth** | Firebase Auth | JWT verification on every protected route |
| **Database** | Firestore Rules | Owner-based access; server-only writes for processing data |
| **Storage** | Storage Rules | File type restrictions (PDF/DOCX/TXT), 5MB max, owner-only access |
| **Headers** | Security Headers | CSP, HSTS, X-Frame-Options, X-Content-Type-Options via `next.config.ts` |
